📖 為什麼需要 Git？
生活例子：小明的論文災難
小明寫論文，每改一版就另存新檔：
論文_final.docx → 論文_final2.docx → 論文_真的最終版.docx → 論文_老師改過.docx

三個月後他完全不知道哪個版本才是最新的，改回去的方法也沒有了。

Git 就是幫你自動管理這些版本的工具，讓你隨時可以「回到過去」。
❌ 沒有 Git 的世界
手動複製貼上、檔名加 _v2 _final _真的最終，出錯了不知道怎麼復原，多人合作直接覆蓋彼此的改動
✅ 有 Git 的世界
每次改動都有記錄，可以跳回任意歷史版本，多人同時改同一個專案也不會衝突
一句話記憶
Git = 程式碼的時光機，GitHub = 放時光機的雲端倉庫
🗂️ Git 的三個區域

小美在超商買東西：先把商品放進籃子（Working Directory）→ 拿到結帳台（Staging Area）→ 收據存檔（Repository）

🖥️ 基本流程指令
# 查看目前狀態 git status # 把改動放進籃子（Staging） git add index.html # 加單一檔案 git add . # 加所有改動 # 正式存檔（Commit） git commit -m "新增首頁標題"

🌿 Branch：平行時空開發
生活例子：小美的食譜實驗
小美有一份基本牛肉麵食譜（main）。她想實驗加辣椒的版本，但又怕失敗毀了原本的配方。於是她影印一份，在影本上實驗（新建 branch）。成功了再把改動合回原本（merge）。
main
穩定版本，隨時可上線
feature/login
阿強在開發登入功能
feature/dark-mode
小美在開發深色模式
🌿 Branch 操作指令
# 建立新分支 git branch feature/login # 切換到分支（現代寫法） git switch feature/login # 建立 + 立刻切換 git switch -c feature/login # 合併分支到 main git switch main git merge feature/login
☁️ GitHub = 雲端倉庫 + 協作平台
Git vs GitHub 的差別
Git 是工具（安裝在你電腦上的程式），GitHub 是平台（存在網路上的倉庫）。
就像 Word 是軟體，Google Drive 是雲端空間 — 兩個是不同東西，但可以搭配使用。
🚀 第一次推上 GitHub
# 綁定遠端倉庫 git remote add origin https://github.com/你的帳號/專案.git # 推送到 GitHub git push -u origin main # 之後每次推送只要 git push
⬇️ 從 GitHub 取得專案
# 第一次抓取（clone） git clone https://github.com/帳號/專案.git # 之後更新（pull） git pull # pull = fetch + merge # 先抓遠端改動，再合併到本機
push vs pull 記憶法
push = 推上去（本機 → 雲端）    pull = 拉下來（雲端 → 本機）
🔀 Pull Request：正式提交改動的流程
生活例子：阿強交報告給老師
阿強寫完報告（在自己的 branch 改好），不直接貼到班級共享資料夾（不直接推到 main），而是先交給老師審查（開 PR）。老師看完沒問題才放進去（merge）。這樣可以確保品質，也留下討論記錄。
1
在自己的 branch 完成功能開發，git push origin feature/login
2
到 GitHub 網頁點「Compare & Pull Request」，寫清楚這次改了什麼
3
組員（或老師）進行 Code Review，留下意見或直接 Approve
4
按下 Merge Pull Request，改動正式進入 main，完成！
為什麼要用 PR，不直接推 main？
避免未完成或有 bug 的程式碼進入主線；留下審查記錄；方便多人協作討論。這是業界標準工作流程。
