概念 01

先搞懂這 5 個名詞

🏢
LINE Developers
開發者後台
另一個網站，跟 LINE App 分開
🗂️
Provider
帳戶容器
你的「公司」，可放多個 Channel
🤖
Channel
機器人帳號
一個 LINE 官方帳號 = 一個 Channel
🔑
Access Token
機器人的鑰匙
程式用這個代表機器人身份發訊
📬
User ID
你的收件地址
指定「把訊息發給誰」的識別碼

Access Token 長這樣

一串很長的英數字，從 LINE Developers 後台「發行」取得。

LINE_ACCESS_TOKEN=xxx...xxx

User ID 長這樣

固定以大寫 U 開頭，後面跟 32 位英數字。這是「你」的 LINE 識別碼。

U1234abcd...（共 33 碼）
⚠️ 常見誤解： LINE Notify 是已停用的舊服務。本系統使用的是 LINE Messaging API，兩者完全不同。

這五個名詞是整個系統的骨架，不用死記，先理解類比就夠了。LINE Developers 是另一個跟 LINE App 完全分開的後台網站。

Provider 像是帳戶容器，Channel 是一個機器人帳號，Access Token 是機器人的鑰匙，User ID 是你的收件地址。

重點要特別說：我們用的是 LINE Messaging API，不是已停用的 LINE Notify——這兩個名字非常像，常被混淆，後面如果有人 Google 到舊文件，記得這點。

設定 02

在 LINE Developers 建立 Channel

  1. 開啟瀏覽器，進入 developers.line.me，用你的 LINE 帳號登入。
  2. 點「Create a new provider」→ 輸入任意名稱（例如：My Assistant）→ 建立。
  3. 點「Create a new channel」→ 選「Messaging API」。
  4. 填寫頻道名稱（任意）、說明（任意）→ 同意條款 → 建立。
  5. 進入剛建立的 Channel → 點選上方「Messaging API」分頁 → 找到「Channel access token」→ 點 Issue 發行 → 複製這串 Token。
  6. 在同一分頁找到「Bot basic ID」→ 用你的 LINE 掃描 QR Code 或搜尋 ID → 把機器人加為好友（必要，否則無法收到訊息）。
你拿到的 Token 長這樣（範例）
LINE_ACCESS_TOKEN=Abcd1234EfghIjkl5678MnopQrst9012UvwxYzab...（共約 172 個字元）
✓ 完成標誌： 你已複製一串很長的 Token，並在 LINE 裡看到機器人帳號的好友。

這個步驟全程在 LINE Developers 網站操作，跟平常用的 LINE App 是完全不同的網站，先確認你打開的是對的地方。

有兩個步驟最容易漏掉：一是 Issue Token 之後要立刻複製，視窗關掉就不見了；二是一定要把機器人加為好友——很多人忘記這步，然後一直收不到訊息，卻以為是程式的問題。

完成標誌：手上有一串 Token，而且 LINE 裡面看得到機器人帳號的好友。兩件事都有才算完成。

設定 03

取得你的 User ID（最常卡這步）

  1. 在 LINE Developers → 你的 Channel → Messaging API → 開啟 Use webhooks。
  2. 到 webhook.site（免費工具）取得一個臨時 URL，貼入 Webhook URL 欄位。
  3. 用你的 LINE 傳任何訊息給機器人帳號。
  4. 在 webhook.site 頁面看到新的請求 → 找到 source.userId 欄位 → 複製其值。
注意： webhook.site 的 URL 是臨時的，只需要用一次取得 User ID 就夠了。

把以下這段話複製後，貼給 Claude、ChatGPT 或任何 AI 助理，它會引導你完成這個步驟：

我在設定 LINE Messaging API 推播系統，需要取得我自己的 LINE User ID（以大寫 U 開頭的 33 碼字串）。我已經在 LINE Developers 建好一個 Messaging API Channel，也把機器人加為好友了。請告訴我最簡單的方法取得 User ID，我使用 Windows，不熟悉架設 webhook 伺服器。
✓ 這個方法適合： 不熟 webhook 設定、只想快速完成的使用者。AI 會根據你的環境給出最簡單的步驟。

User ID 是最多人卡住的地方。問題在於：User ID 不會直接顯示在後台，需要讓機器人「接到你傳的訊息」，才能知道你的 ID 是什麼。

方法 A：用 webhook.site，完全免費，不用架伺服器。開啟網站拿到臨時 URL，貼進 LINE Developers 的 Webhook URL 欄位，用自己的 LINE 傳一則訊息給機器人，就能在 webhook.site 看到 source.userId。

不熟 webhook 的話，直接用方法 B：複製上面的 AI 提示語，丟給 Claude 或 ChatGPT，它們會一步一步帶你。兩種方法都很快，選一個做就好。

.env 是什麼

存放私密設定的檔案。程式會自動讀取這裡的 Token 和 User ID。

⚠️ 這個檔案不要截圖、不要貼到聊天室、不要上傳 GitHub。

設定 04

把 Token 和 User ID 填進 .env

先在 PowerShell 複製 .env.example：

cd C:\path\to\my-assistant\line-notify Copy-Item .env.example .env

然後用文字編輯器（記事本）開啟 .env，填入：

# 在下方填入你的資料，不要加引號 LINE_USER_ID=（等待輸入） LINE_ACCESS_TOKEN=（等待輸入）
⚠️ 常見格式錯誤： 不要在 = 後加引號（如 "Uxxxxxx"）；變數名稱必須是 LINE_ACCESS_TOKEN，不是 LINE_CHANNEL_ACCESS_TOKEN。

.env 就是「環境設定檔」，是程式的機密保險箱，Token 和 User ID 都存在這裡，程式每次執行時會自動讀取。

三個容易犯的錯誤：一、等號後面不要加引號；二、變數名稱必須完全正確，LINE_ACCESS_TOKEN 不是 LINE_CHANNEL_ACCESS_TOKEN；三、這個檔案不要截圖、不要上傳 GitHub、不要貼給別人，裡面有你的私密金鑰。

這頁有一個即時驗證器，可以直接貼上 User ID 和 Token 確認格式正確，不用靠猜測。

使用 06

每天改這一個檔案：schedule.json

events（行程）

date

格式：YYYY-MM-DD（例如 2026-05-07）

priority

high = 紅色重要提醒；normal = 一般提醒

done

完成後改成 true，不會再顯示在早報

tasks（待辦）

status

todo 待開始 / in_progress 進行中 / done 完成

done_at

完成日期（YYYY-MM-DD）；未完成填 null

設定完成後，日常使用主要就是改這一個 JSON 檔案。行程放在 events 陣列，待辦放在 tasks 陣列，改完後下一次推播的內容就會自動更新。

右邊的 JSON 生成器可以快速產生格式正確的片段，填好欄位按生成，複製後貼進 schedule.json 就好。

如果不確定 JSON 格式是否正確，用 Python 的 json.tool 驗證——格式正確會輸出格式化的內容，格式錯誤會直接告訴你哪一行有問題。

設定 07

Windows 工作排程器設定

例如：C:\Users\SunLee\projects\my-assistant

以系統管理員身分執行 PowerShell，貼入以下指令：

$Project = "C:\path\to\my-assistant" $Python = "$Project\.venv\Scripts\python.exe" $Script = "$Project\line-notify\digest.py" $WorkDir = "$Project\line-notify" $MornA = New-ScheduledTaskAction -Execute $Python -Argument "`"$Script`" --mode morning" -WorkingDirectory $WorkDir $MornT = New-ScheduledTaskTrigger -Daily -At 8:00AM Register-ScheduledTask -TaskName "line-notify-morning" -Action $MornA -Trigger $MornT $EveA = New-ScheduledTaskAction -Execute $Python -Argument "`"$Script`" --mode evening" -WorkingDirectory $WorkDir $EveT = New-ScheduledTaskTrigger -Daily -At 8:00PM Register-ScheduledTask -TaskName "line-notify-evening" -Action $EveA -Trigger $EveT

測試排程是否有效：

Start-ScheduledTask -TaskName "line-notify-morning"
  1. 開啟「工作排程器」（搜尋列輸入 Task Scheduler）
  2. 點「建立基本工作」→ 名稱填 line-notify-morning
  3. 觸發程序選「每天」→ 時間設 08:00
  4. 動作選「啟動程式」→ 程式填：C:\path\to\my-assistant\.venv\Scripts\python.exe
  5. 新增引數填：C:\path\to\my-assistant\line-notify\digest.py --mode morning
  6. 開始位置填：C:\path\to\my-assistant\line-notify
  7. 重複步驟 2-6，名稱改 line-notify-evening，時間改 20:00，引數改 --mode evening

到這一步代表手動推播已經成功了——這是前提，手動沒成功就不要先來設排程，否則排程設好了也沒用，反而難排查問題。

方法 A 的 PowerShell 一行指令是最快的做法。記得先填好上面的路徑輸入框，後面的指令會自動替換，不用手動改。執行時要用系統管理員身分開啟 PowerShell。

設定完成後，電腦每天 08:00 和 20:00 會自動觸發，不用手動。

排錯 08

卡關了？對照這裡

沒收到LINE 沒收到訊息，但指令沒報錯▶
確認機器人帳號已加為好友。確認 .env 的 LINE_USER_ID 是 source.userId，不是 channel id 或 bot id。
401LINE 回傳 401 Unauthorized▶
Token 錯誤或過期。回 LINE Developers → Messaging API → 重新發行 Channel access token → 更新 .env 的 LINE_ACCESS_TOKEN。
400LINE 回傳 400 Bad Request▶
User ID 格式不對或機器人帳號未加好友。確認 LINE_USER_ID 以大寫 U 開頭、共 33 碼。
缺模組ModuleNotFoundError: No module named 'httpx'▶
cd C:\path\to\my-assistant .\.venv\Scripts\python.exe -m pip install httpx
JSONjson.decoder.JSONDecodeError▶
schedule.json 格式壞掉了。執行驗證：
..\.venv\Scripts\python.exe -m json.tool .\schedule.json

常見錯誤：最後一筆後多了逗號、null/true/false 加了引號。

排程失敗手動成功，但工作排程器沒有發▶
檢查：
① 「程式或指令碼」必須是 .venv\Scripts\python.exe 的完整路徑
② 「開始位置」必須是 line-notify 資料夾
③ 電腦在排程時間必須開機
Get-ScheduledTaskInfo -TaskName "line-notify-morning"

LastTaskResult 不是 0 代表上次執行失敗。

路徑路徑包含空白（如 C:\Users\Sun Lee）▶
PowerShell 用引號包住完整路徑：
& "C:\Users\Sun Lee\projects\my-assistant\.venv\Scripts\python.exe" "C:\Users\Sun Lee\projects\my-assistant\line-notify\digest.py" --mode morning
缺變數缺少環境變數：LINE_USER_ID, LINE_ACCESS_TOKEN▶
.env 放錯位置，或變數名稱打錯。確認：
① 檔案路徑：line-notify\.env（不是 my-assistant\.env）
② 變數名稱是 LINE_ACCESS_TOKEN，不是 LINE_CHANNEL_ACCESS_TOKEN
③ = 前後沒有多餘空格或引號

如果出問題，診斷順序很重要：先手動跑一次 digest.py，確認手動成功；手動成功了才去排查自動排程的問題。

最常見的三個錯誤：401 = Token 過期，去後台重新發行；400 = User ID 格式錯或沒加好友；沒收到但沒報錯 = User ID 填的是機器人 ID，要填自己的 source.userId。
