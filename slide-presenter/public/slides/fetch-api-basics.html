為什麼需要 API？

蝦皮有幾百萬筆商品，不可能塞進前端

✅ API 的解法
資料放在 資料庫，前端透過 API 請求
API = 程式跟程式之間的「服務生」
角色對應到程式
👤 客人（你）你寫的程式（前端）
🧑‍💼 服務生API
👨‍🍳 廚房伺服器（Server）
📦 食材倉庫資料庫
API 全名
Application Programming Interface — 應用程式介面，讓不同程式互相溝通的規範
HTTP 長什麼樣？
每次 fetch 就是送一封信、收一封回信
📤 你送出去的（Request）
GET 請求
GET /api/livejs/v1/customer/sun90/products Host: livejs-api.hexschool.io Accept: */*
① 方法 + 路徑 — 「我要做什麼、對什麼資源」
② Host — 走進哪一家餐廳
③ GET 不帶 body — 只問，不送東西
📥 伺服器回你的（Response）
成功回應
200 OK Content-Type: application/json // body（真正的資料） { "products": [ { "title": "沙發", "price": 6600 } ] }
① 200 OK — 狀態碼，告訴你成不成功
② Content-Type — 裡面是 JSON 格式
③ body — 空一行後才是真正的資料
HTTP 五個方法
告訴伺服器「你想對資料做什麼事」
方法對資料做什麼餐廳比喻你的作業
GET讀取（只看，不動）「給我看菜單」getProducts / getCart
POST新增「我要下單：拉麵+煎餃」addToCart
PATCH修改部分欄位「炒飯改成炒麵就好」updateCartItem
PUT整筆取代「整單重來，換成炒飯」（作業沒用到）
DELETE刪除「取消訂單」removeCartItem / clearCart
PATCH vs PUT 的差別
改手機號碼：
PUT → 10 個欄位全部重送，server 整份覆蓋
PATCH → 只送「phone」一個欄位，其他不動
✅ 實務上幾乎都用 PATCH，省又安全
DELETE 兩種用法
刪特定一筆：DELETE /carts/:id
刪全部：DELETE /carts

差別只在網址有沒有帶 id
RESTful API
一種 API 設計風格，讓前後端溝通有統一規則
核心觀念
網址只描述「資源是什麼」，動作交給 HTTP 方法表達
❌ 非 RESTful（亂的）
// 動作寫在網址裡，沒有規律 GET /getProducts GET /fetchCart POST /addToCart POST /deleteCartItem POST /updateCart
✅ RESTful（你的作業 LiveJS）
// 同一個網址，方法決定行為 GET /carts ← 看購物車 POST /carts ← 加商品 PATCH /carts ← 改數量 DELETE /carts ← 清空 DELETE /carts/:id ← 刪一筆
一句話記住
同個網址 + 不同方法 = 不同行為，才是好的 API 設計
HTTP 狀態碼
伺服器回信時帶的三位數字，記第一個數字就好
開頭意思餐廳比喻
1xx收到了，處理中「點餐單收到，廚房準備中」
2xx成功 ✓「您的餐點來了」
3xx去別的地方「我們搬家了，請去隔壁」
4xx你的錯誤「你點的菜單上沒有 / 沒帶會員卡」
5xx伺服器爆炸「廚房失火，今天不出餐」
必記清單
200 成功   201 新增成功（POST 常回）
400 請求格式錯   401 沒登入
403 沒有權限   404 找不到
500 Server 自己爆了
最簡單的記法
4 開頭 = 你的錯
5 開頭 = 別人的錯
同步 vs 非同步
為什麼 fetch 不能像一般程式一樣等它跑完？
同步（你以前寫的程式）
console.log('1'); console.log('2'); console.log('3'); // 永遠按順序：1, 2, 3
同步的問題
如果 fetch 是同步的，等 API 回應的 2 秒內整個瀏覽器凍住——畫面不動、按鈕沒反應
非同步（fetch 的運作方式）
號碼牌機制
發出 fetch →「給你號碼牌，先做別的」
JavaScript 繼續跑其他程式
2 秒後資料到 →「叫號了，來拿資料」
直接印 fetch 的結果
const result = fetch('...'); console.log(result); // Promise { <pending> } // ↑ 這就是號碼牌，資料還在路上
Promise & await
Promise 是號碼牌，await 是去兌換號碼牌
Promise 是什麼
一個「等一下會裝著資料的盒子」

看到 Promise = 資料還沒到，等一下會到
要用 await 打開盒子拿值
❌ 沒加 await
const result = fetch('...'); // Promise { <pending> } ← 號碼牌
✅ 加了 await
const result = await fetch('...'); // Response { status: 200, ok: true } // ↑ 真正的 response 物件，不是 pending 了
⚠️ 直接用 await 會報錯
await 只能在 async 函式 裡用
直接在最外層寫會：
SyntaxError: await is only valid in async functions
三個規則
1. 看到 Promise → 用 await 打開    2. 用了 await → 函式前加 async    3. 宣告 ≠ 執行 → 記得呼叫函式
async / await 完整寫法
fetch 跟 json 兩兄弟，一個都不能少 await
fetch GET 標準 pattern
async function getProducts() { // 第一個 await：等「收到回信」（信封到了） const response = await fetch(`${BASE_URL}/api/.../products`); // 第二個 await：等「打開信封、解析 JSON 字串」 const data = await response.json(); return data.products; // ← 路徑看 API 文件決定 } // 呼叫時也要 await（async function 回傳 Promise） const products = await getProducts();
❌ 常見錯誤一：漏第二個 await
const data = response.json()
→ 印出來是 Promise pending，資料拿不到
❌ 常見錯誤二：沒有 return
函式裡只寫 console.log(data)
→ 外面拿到 undefined
console.log ≠ return！
response 物件裡有什麼？
fetch 給你的是整封信（信封），.json() 才是打開信封
response 的欄位
const response = await fetch(url); response.status // 200 / 404 / 500... 狀態碼數字 response.ok // true（2xx） 或 false（其他） response.statusText // 'OK' / 'Not Found'... 文字說明 await response.json() // 把 body 的 JSON 字串解析成 JS 物件
response.ok 最重要
2xx 時 = true
4xx / 5xx 時 = false

等同於：response.status >= 200 && response.status < 300
但用 ok 更簡潔
信封 vs 包裹比喻
response = 整封信（信封）
response.status = 信封上的狀態
response.json() = 打開信封、拿出包裹裡的東西
錯誤處理
fetch 不會自動幫你報錯，要自己寫檢查
⚠️ 關鍵觀念
fetch 只有在「網路斷線、連不上 server」才自動丟錯誤。Server 回 404、500 時，fetch 不報錯，只是靜靜給你那個失敗的 response。
getProductsSafe() — 完整錯誤處理
async function getProductsSafe() { try { const response = await fetch(url); if (!response.ok) { // response.ok = false 時，主動宣告失敗 throw new Error(`HTTP error! status: ${response.status}`); } const data = await response.json(); return { success: true, data: data.products }; } catch (error) { // throw 丟出來的 Error，在這裡被接住 return { success: false, error: error.message }; } }
throw new Error 做什麼
建立一個錯誤物件，然後「丟出去」讓 catch 接住。程式遇到 throw 就立刻跳到 catch，後面的程式不執行。
回傳格式統一
成功：{ success: true, data: [...] }
失敗：{ success: false, error: '訊息' }

呼叫端只要檢查 success 就知道狀態
fetch POST 請求
GET 像明信片（只有地址），POST 像包裹（地址 + 內容）
addToCart() — POST 標準 pattern
async function addToCart(productId, quantity) { const response = await fetch(`${BASE_URL}/.../carts`, { method: 'POST', // ① 不寫的話 fetch 預設 GET headers: { 'Content-Type': 'application/json' // ② 告訴 server「我寄的是 JSON」 }, body: JSON.stringify({ // ③ 物件要壓成字串才能寄 data: { productId, quantity } // ④ LiveJS 規定要包 data 層 }) }); return await response.json(); }
為什麼要 JSON.stringify？
網路只能傳字串，不能傳 JS 物件。
JSON.stringify 把物件「壓平」成字串才能寄。
server 收到後自己再解析回物件。
body 格式看 API 文件
不是每個 API 都包 { data: ... }，格式由每個 API 自己規定。呼叫前一定要看 Swagger 文件確認。
環境變數 .env
為什麼 API Key 不能寫死在程式碼裡？
❌ 寫死的危險
const API_KEY = "sk-abc123xyz"

只要你 push 到 GitHub，任何人都能看到你的 key，被盜用就慘了
✅ 放在 .env
建立 .env 檔案（不要 commit！）
API_PATH=你的path
API_KEY=你的key

.gitignore 已經把 .env 排除了
程式碼讀取方式
// 載入 .env require('dotenv').config(); // 用 process.env 讀取 const API_PATH = process.env.API_PATH; const API_KEY = process.env.API_KEY; // 用在 URL 組合 const url = `${BASE_URL}/api/.../${API_PATH}/products`;
為什麼叫「環境變數」
不同環境（開發/測試/上線）可以有不同的 key，程式碼不用改，只換 .env 檔案就好
